# v0.7 회계관리 고도화 계획

기준 버전: `v0.6.9-release-verification-v07-handover`

문서 상태: v0.6 유통관리/송장/배송 구현 결과를 바탕으로 v0.7에서 다룰 회계관리 고도화 범위와 제외 범위를 정리한 계획 문서다. v0.7 기능은 아직 구현되지 않았으며, 이 문서는 v0.7.0 이후 작업 기준으로 사용한다.

## 목표

v0.7은 현재 단일 회계 내역 중심 구조를 확장해 매입, 매출, 환불, 배송비, 택배비, 정산, 마감, 회계 리포트를 실무 ERP 포트폴리오 수준으로 축소 구현하는 것을 목표로 한다.

## v0.6에서 넘어온 데이터 후보

- 주문 매출: 주문 생성, 결제 완료, 주문 상태 변경 기준 매출 인식 후보.
- 환불: 결제 취소/환불, 반품 승인, 주문 `REFUNDED` 상태 기준 환불 인식 후보.
- 배송비 매출: 주문 배송비 또는 배송 정책 기반 고객 청구 배송비 후보.
- 반품 배송비: `return_shipment_infos.shipping_fee`, `fee_payer` 기준 고객/회사 부담 구분 후보.
- 택배비 매입: `shipping_methods.default_fee`, 택배사/배송 방법 기준 내부 비용 후보.
- 출고 기준 매출 확정: 출고 검수 완료 또는 배송중 전환 시점 기준 매출 확정 후보.
- 송장/라벨/추적 비용: 실제 택배사 API/라벨 출력 비용은 후속 외부 연동 이후 연결 후보.

- 관리자 회계 화면, 정산 화면, 회계 리포트 화면 후보.

### v0.7.x 공통 계획 항목

각 v0.7.x 작업은 아래 기준을 체크리스트에 기록한다.

- 목적: 해당 버전에서 해결할 회계/정산 문제.
- 포함 범위: 실제 구현 또는 문서화 대상.
- 제외 범위: v0.7 후속 또는 v0.8 이후로 넘기는 항목.
- DB 변경 후보: 테이블, 컬럼, FK, 인덱스, 상태값.
- API 변경 후보: 경로, method, request/response DTO, 권한.
- 프론트 화면 변경 후보: 경로, 테이블 컬럼, 필터, 버튼, 빈/오류 상태.
- 권한 정책: role 1차 접근과 permission code 세부 실행 기준.
- 감사 로그 기록 기준: 작업 유형, 대상, 변경 전/후, metadata.
- 완료 조건: 구현/문서/검증/태그 기준.
- 검증 기준: backend test, frontend lint/build, 문서 링크/인코딩 확인.
- 문서화/인코딩 확인 기준: 모든 수정 MD는 UTF-8, 한글 깨짐 없음, 표/링크/코드블록 정상.

기존 도메인 연결 후보:

- `orders`: 주문 매출, 주문 취소, 주문 상태 기준 참조.
- `order_items`: 상품별/SKU별 매출과 수익성 후보.
- `payments`: 결제 완료, 취소, 환불 참조.
- `refunds`: 별도 환불 도메인이 분리된 경우 환불 거래 참조.
- `return_requests`: 반품 승인/입고/환불과 반품 배송비 참조.
- `shipments`: 송장 발급, 배송중, 배송완료, 배송비 비용 참조.
- `outbound_orders`: 출고 피킹/출고 완료 기준 매출 확정 후보.

## 권한 정책

기존 role 기반 1차 접근은 유지한다. `/api/admin/**`는 `MANAGER`, `ADMIN`, `SUPER_ADMIN` 중 하나여야 접근할 수 있고, 회계 실행 권한은 permission code로 세분화한다.

| 작업 | permission 후보 |
| --- | --- |
| 회계 조회 | `ACCOUNTING_READ` |
| 회계 거래 관리 | `ACCOUNTING_MANAGE` |
| 정산 생성/마감 | `SETTLEMENT_MANAGE`, `ACCOUNTING_CLOSE` |
| 회계 리포트 조회 | `ACCOUNTING_REPORT_READ` 또는 `ACCOUNTING_READ` |
| 결제 환불 | `PAYMENT_REFUND` |
| 배송비/택배비 비용 관리 | `SHIPPING_COST_MANAGE` |
| 반품 배송비 회계 처리 | `RETURN_FEE_MANAGE` |
| 회계/정산 설정 변경 | `SETTINGS_MANAGE` |
| 감사 로그 조회 | `AUDIT_LOG_READ` |

신규 permission 후보:

- `ACCOUNTING_MANAGE`
- `SETTLEMENT_MANAGE`
- `ACCOUNTING_REPORT_READ`
- `SHIPPING_COST_MANAGE`
- `RETURN_FEE_MANAGE`

`SUPER_ADMIN`은 전체 허용한다. `ADMIN`은 운영/회계 관리 권한을 permission group으로 부여받는 구조를 유지한다. `MANAGER`는 기본적으로 조회 중심 권한만 부여하고, 정산 마감/거래 보정/비용 등록은 제한한다.

## 문서화와 인코딩 기준

- v0.7.0 단계에서는 구현되지 않은 항목을 “후보”, “계획”, “이관”으로 표현한다.
- 구현 완료된 v0.6 이하 기능만 “완료”라고 표현한다.
- 문서 경로는 현재 구조인 `docs/architecture`, `docs/plans`, `docs/operations`, `docs/ai` 기준으로 작성한다.
- 오래된 루트 문서 경로를 새로 추가하지 않고, 현재 canonical 경로만 사용한다.
- 모든 수정 MD는 UTF-8 기준으로 저장하고 한글 깨짐, BOM 이상 문자, 표 깨짐, 코드블록 미종료, 상대 링크 오류를 확인한다.

## v0.7.3 진행 상태

결제 환불과 반품 승인 환불을 `REFUND` 회계 거래로 반영하고, 반품 배송비 부담 주체에 따라 `RETURN_FEE` 거래를 `INCOME` 또는 `EXPENSE`로 기록하는 기반을 추가한다.

- 결제 환불 거래: `referenceType=PAYMENT`, `type=REFUND`.
- 반품 환불 거래: `referenceType=RETURN`, `type=REFUND`.
- 반품 배송비 거래: `referenceType=RETURN`, `type=RETURN_FEE`.
- 고객 부담 반품 배송비는 수익, 회사 부담 반품 배송비는 비용으로 처리한다.
- 부분 환불, 부분 반품, 실제 청구/차감 정산 자동화는 후속 버전으로 이관한다.

## v0.7.4 진행 상태

택배사/배송 방법 기준 내부 택배비 매입 비용을 `shipping_cost_entries`와 `SHIPPING_COST` 회계 거래로 반영하는 기반을 추가한다.

- 배송비 비용 거래: `referenceType=SHIPMENT`, `type=SHIPPING_COST`.
- 비용 금액은 `shipping_methods.default_fee`를 사용한다.
- 고객 청구 배송비 매출은 주문 배송비 컬럼이 없어 이번 버전에서는 0원 후보로 보관하고, 후속 배송비 매출 설계로 이관한다.
- 정산 배치 연결과 마감 반영은 v0.7.5 이후에서 확장한다.

## v0.7.5 진행 상태

기간별 회계 거래를 정산 배치로 묶고 마감 상태를 관리하는 기반을 추가한다.
