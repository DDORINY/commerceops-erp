# 문서화 운영 기준

## 목표

문서는 구현을 설명하는 사후 보고서가 아니라 범위 합의, 완료 판정, 장애 대응에 쓰는 도구다. 문서와 코드가 다르면 실제 동작은 코드가 결정하지만, 그 차이는 결함으로 보고 같은 작업에서 고친다.

## 반드시 유지할 문서

| 문서 | 포함할 내용 | 책임 시점 |
| --- | --- | --- |
| 제품 로드맵 | 버전 목표, 시작/종료점, 제외 범위 | 분기/버전 계획 시 |
| 기능 명세 | 사용자, 시나리오, 정책, 상태, API, 완료 조건 | 개발 시작 전 |
| API 계약 | 요청/응답, 권한, 오류, 멱등성, 예시 | API 변경과 동시에 |
| 데이터 사전/ERD | 필드 의미, 관계, 제약, 보존, 개인정보 | DB 변경과 동시에 |
| 상태 전이표 | 허용 전이, 수행 주체, 부수 효과, 실패 처리 | 상태형 도메인 변경 시 |
| ADR | 선택지, 결정, 이유, 결과 | 중요한 기술 결정 시 |
| 테스트 전략 | 위험, 테스트 계층, 핵심 E2E, 데이터 | 버전 계획 시 |
| 운영 런북 | 배포, 롤백, 장애, 백업/복구, 배치 재처리 | 운영 기능 도입 시 |
| 보안/개인정보 | 권한, 위협 모델, 보존/파기, 비밀 관리 | 출시 전과 정책 변경 시 |
| 릴리스 노트 | 변경, 마이그레이션, 알려진 문제, 롤백 | 매 릴리스 |

## 특히 더 남겨야 할 문서

현재 프로젝트에서 우선순위가 높은 것은 다음과 같다.

1. **주문·배송·반품·결제 상태 전이표**: 잘못된 전이가 재고와 금액 오류로 이어진다.
2. **재고 원장 정책**: 언제 예약/차감/복구하고 중복 처리를 어떻게 막는지 정의한다.
3. **결제·환불 멱등성 및 대사 설계**: 웹훅 중복, 타임아웃, 실패 재처리를 다룬다.
4. **회계 이벤트 매핑표**: 업무 이벤트마다 어떤 차변/대변 전표가 생성되는지 남긴다.
5. **권한 매트릭스**: 역할별 화면, API, 데이터 범위와 승인 권한을 표로 관리한다.
6. **데이터 분류/보존표**: 개인정보, 결제 식별자, 로그의 보존 기간과 파기 방법을 적는다.
7. **배포·롤백·DB 복구 런북**: 담당자가 바뀌어도 서비스를 복구할 수 있어야 한다.
8. **비기능 요구사항**: 목표 응답 시간, 동시 사용자, 가용성, RPO/RTO, 비용 한도를 수치로 정한다.

## 기능 개발 문서 흐름

```text
요구/문제 정의
→ 기능 명세 작성
→ 상태/API/DB 영향 검토
→ 필요한 ADR 작성
→ 구현과 테스트
→ API/DB/운영 문서 동기화
→ 릴리스 체크리스트와 릴리스 노트
```

## 기능 명세 최소 항목

- 문제와 목표 사용자
- 사용자 시나리오와 운영자 시나리오
- 범위와 명시적인 제외 범위
- 비즈니스 규칙과 상태 전이
- 권한과 개인정보
- API/DB/UI 변경
- 금액·재고에 대한 트랜잭션과 멱등성
- 오류, 재시도, 취소, 복구 방식
- 수용 기준과 테스트 시나리오
- 지표, 로그, 알림
- 배포, 마이그레이션, 롤백

템플릿은 [FEATURE_SPEC_TEMPLATE.md](./templates/FEATURE_SPEC_TEMPLATE.md)를 사용한다.

## ADR 작성 대상

다음 결정은 짧더라도 ADR을 남긴다.

- 결제/재고/회계 트랜잭션 경계
- 주문·배송·반품 상태 모델
- 인증 토큰 저장/갱신 방식
- DB 마이그레이션 도구
- 파일 저장소, 메시지 큐, 검색 엔진 등 외부 인프라
- 멀티테넌시의 데이터 격리 방식
- 모놀리스 모듈을 별도 서비스로 분리하는 결정

ADR은 결정을 바꾸지 못하게 하는 문서가 아니다. 바뀌면 이전 ADR을 삭제하지 않고 새 ADR에서 대체 관계를 기록한다.

## 문서 갱신 규칙

- PR/작업 단위에서 코드와 문서를 함께 변경한다.
- 예시 JSON과 enum은 가능한 한 테스트나 OpenAPI에서 자동 생성한다.
- 계획과 현재 상태를 섞지 않는다. 예정은 `⬜`, 구현 중은 `🟡`, 검증 완료만 `✅`로 표시한다.
- 날짜만 오래된 문서보다, 마지막 검증 커밋/버전과 담당 영역이 없는 문서를 더 위험하게 본다.
- 임시 artifact와 대화 링크만 원본으로 두지 않는다. 중요한 결정과 범위는 저장소 Markdown으로 옮긴다.
- 비밀번호, 토큰, 실제 고객 데이터는 문서 예시에 넣지 않는다.

## 권장 추가 디렉터리

프로젝트가 커지면 다음 구조로 확장한다.

```text
docs/
├── product/        # PRD, 사용자 흐름, 정책
├── features/       # 기능별 명세
├── architecture/   # 시스템 구성, 통합, 비기능 요구사항
├── adr/            # 기술 의사결정
├── api/            # OpenAPI와 연동 가이드
├── data/           # ERD, 데이터 사전, 보존 정책
├── operations/     # 배포, 장애, 백업, 배치 런북
├── security/       # 위협 모델, 권한, 개인정보
├── releases/       # 버전별 체크리스트와 릴리스 노트
└── templates/      # 재사용 템플릿
```

지금 당장 빈 폴더를 모두 만들 필요는 없다. 첫 문서가 생길 때 해당 디렉터리를 추가한다.
