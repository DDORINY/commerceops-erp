# CommerceOps ERP 제품 로드맵

## 제품 목표

CommerceOps ERP는 상품 판매 화면만 제공하는 쇼핑몰이 아니라 다음 흐름을 하나의 데이터로 연결하는 운영 서비스다.

```text
상품/조달 → 재고 → 주문 → 결제 → 배송 → 반품/환불
                              ↓
                     회계/정산 → 분석/자동화
```

장기적으로는 한 사업자의 내부 시스템을 넘어 여러 판매 조직이 사용할 수 있는 커머스 운영 플랫폼까지 확장할 수 있게 설계한다.

## 로드맵 원칙

1. 새 모듈보다 먼저 기존 주문 생명주기를 끝까지 신뢰할 수 있게 만든다.
2. 재고·결제·환불·회계처럼 금액과 수량을 바꾸는 작업은 트랜잭션, 멱등성, 감사 로그를 필수로 한다.
3. 화면, API, DB, 테스트, 운영 문서가 모두 완료되어야 기능을 완료로 본다.
4. 버전은 기능 개수보다 사용자에게 전달 가능한 하나의 운영 흐름을 기준으로 자른다.
5. 운영 준비 없이 기능만 늘리지 않는다. 보안·관측성·백업·배포 자동화도 제품 범위다.

## 핵심 도메인 상태 흐름

### 주문/배송

```text
PENDING → PAID → PREPARING → SHIPPING → COMPLETED
    └────────────── 조건에 따라 CANCELLED
```

배송은 `PREPARING`에서 생성하고 송장 등록 시 `IN_TRANSIT`, 배송 완료 시 `DELIVERED`로 맞춘다. 주문 상태와 배송 상태가 서로 어긋나지 않도록 단일 서비스 또는 도메인 이벤트로 처리한다.

### 반품/환불

```text
REQUESTED → APPROVED → 환불 실행 → 재고 복구 → 회계 반제 → REFUNDED
          └→ REJECTED
```

현재 구현처럼 반품 승인만으로 모든 후속 처리를 한 번에 끝내기보다, 실제 서비스에서는 수거·검수·환불 상태를 분리해야 한다. 부분 반품과 실패 재처리도 고려한다.

### 회계

```text
결제/환불/입고 이벤트 → 전표 생성 → 차변/대변 검증 → 확정 → 기간 마감 → 보고서
```

주문 데이터와 회계 전표는 수정으로 덮지 않고 원거래와 취소/반제 거래를 남겨 감사 가능하게 한다.

## 우선순위 판단 기준

기능 후보는 아래 순서로 평가한다.

1. 금액·재고·개인정보 오류를 막는 안정성
2. 주문부터 사후 처리까지 끊어진 운영 흐름
3. 실제 서비스 출시에 필요한 법무·보안·운영 요건
4. 반복 업무를 줄이는 자동화
5. 매출 성장과 사용자 경험
6. 새로운 ERP 모듈 및 SaaS 확장

## 로드맵 변경 규칙

- 버전 중간에 새 기능을 넣으려면 같은 버전에서 뺄 항목 또는 일정 영향을 함께 기록한다.
- 상태 전이, 결제, 재고, 회계 모델 변경은 ADR을 작성한다.
- API와 DB를 바꾸는 기능은 구현 PR에서 해당 문서를 같이 갱신한다.
- 버전 종료 시 미완료 항목을 조용히 넘기지 않고 다음 버전으로 이동한 이유를 릴리스 노트에 남긴다.
