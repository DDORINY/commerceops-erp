# v0.5 재고관리/생산 입고/바코드 자동화 계획

기준 버전: `v0.5.0-inventory-production-barcode-plan`

문서 상태: v0.5.0에서 확정한 계획 문서다. v0.5.1부터 기능 구현을 시작하며, 이 문서의 범위를 기준으로 체크리스트와 구현 범위를 관리한다.

## 목표

v0.5에서는 상품, 옵션, 주문, 창고 기반 위에 실무 ERP의 재고 운영 흐름을 축소 구현한다. 상품을 매입하거나 생산해서 입고하고, 재고가 자동 증가하며, SKU와 바코드를 기준으로 입고/출고/이동/조정/실사 이력을 추적할 수 있는 운영 기반을 만든다.

최종 목표는 v0.6 유통관리에서 출고 지시, 송장번호 생성, 송장 출력, 배송 상태 추적으로 자연스럽게 이어질 수 있는 재고/출고 기반을 마련하는 것이다.

## 기존 구조 기준

- 상품: `Product`는 `productCode`, 옵션 JSON, `stockQuantity`, `safetyStockQuantity`, 판매/전시 상태를 보유한다.
- 재고: `InventoryLog`는 상품 기준 입고/조정 이력을 기록한다.
- 창고: `Warehouse`, `WarehouseStock`, `StockReservation`, `StockTransfer`로 창고별 재고, 예약, 이동 기반이 있다.
- 주문/배송: 주문, 주문 상품, 배송 상태는 구현되어 있으나 바코드 기반 출고 검수와 송장 자동화는 v0.6 범위로 이관한다.
- 권한: 기존 role 1차 접근과 permission code 세부 실행 정책을 유지한다.
- 감사 로그: 재고 변경, 생산 입고, 바코드 생성/출력, 실사 조정은 audit log 기록 후보로 둔다.

## SKU/바코드 정책

- v0.5.1에서는 SKU를 상품 또는 옵션 조합 단위 재고 기준으로 도입한다.
- 기존 `Product.productCode`는 상품 마스터 코드로 유지하고, SKU는 재고/입출고/바코드 기준 코드로 분리한다.
- 옵션이 없는 상품은 상품당 기본 SKU 1개를 만들 수 있다.
- 옵션이 있는 상품은 옵션 조합별 SKU 생성을 기본 방향으로 둔다.
- SKU code는 시스템 자동 생성을 기본으로 하고, 관리자 수동 입력은 중복 검증 후 허용 후보로 둔다.
- 바코드는 SKU별 1개 기본 바코드를 우선한다.
- 바코드 값은 자동 생성이 기본이며, 기존 바코드가 있는 경우 수동 입력도 허용 후보로 둔다.
- 바코드는 unique 제약으로 중복을 방지한다.
- 바코드 라벨 출력은 HTML print 또는 PDF 후보로 시작한다.
- 실제 프린터 드라이버/스캐너 SDK 연동은 제외한다.

## 생산 입고 정책

- 생산 주문 생성 후 생산 상태를 관리한다.
- 생산 상태 후보는 `PLANNED`, `IN_PROGRESS`, `COMPLETED`, `CANCELLED`다.
- 생산 완료 시 완료 수량만큼 SKU/상품 재고를 증가시킨다.
- 생산 완료 시 `inventory_movements`에 `PRODUCTION_RECEIPT` 유형 이력을 생성한다.
- 생산 입고 감사 로그를 기록한다.
- 생산 완료 후 바코드 라벨 출력 여부를 선택할 수 있는 흐름을 후보로 둔다.
- BOM, 자재 차감, 원가 계산은 v0.5 제외 또는 v0.7 회계/원가 후보로 이관한다.

## 재고 이동/실사/조정 정책

- `inventory_movements`를 v0.5 재고 변경 이력의 중심 모델로 둔다.
- movement type 후보는 `INBOUND`, `OUTBOUND`, `ADJUSTMENT`, `TRANSFER`, `LOCATION_MOVE`, `PRODUCTION_RECEIPT`, `RETURN_INBOUND`, `STOCK_COUNT_ADJUSTMENT`다.
- 창고 간 이동은 기존 `stock_transfers`와의 역할 분리를 검토한다. v0.5에서는 기존 기능을 유지하면서 movement 이력과 연결하는 방향을 우선한다.
- 재고 실사는 세션 시작, 실사 수량 입력, 차이 계산, 완료/조정 순서로 진행한다.
- 실사 완료 시 차이 수량이 있는 품목만 조정 movement를 생성한다.
- 실사 조정은 감사 로그 대상으로 둔다.

## 권한 정책

기존 permission:
- `INVENTORY_READ`
- `INVENTORY_WRITE`
- `WAREHOUSE_MANAGE`
- `PRODUCT_READ`
- `PRODUCT_WRITE`
- `AUDIT_LOG_READ`

v0.5 신규 후보 permission:
- `SKU_MANAGE`
- `BARCODE_MANAGE`
- `PRODUCTION_MANAGE`
- `STOCK_COUNT_MANAGE`
- `STOCK_MOVEMENT_MANAGE`

정책:
- 재고 조회: `INVENTORY_READ`
- 입고/조정/이동: `INVENTORY_WRITE` 또는 `STOCK_MOVEMENT_MANAGE`
- 창고/로케이션: `WAREHOUSE_MANAGE`
- 생산 입고: `PRODUCTION_MANAGE`
- 바코드 생성/출력: `BARCODE_MANAGE`
- 실사: `STOCK_COUNT_MANAGE`
- SKU 관리: `SKU_MANAGE`

- SKU 생성/수정/활성 변경

- 생산 주문 생성/수정/완료
- 생산 입고 처리
- 입고 처리
- 출고 처리
- 재고 조정

- 위치 이동
- 재고 실사 시작/완료

## 검증 기준

v0.5.0은 문서 계획 버전이므로 코드 변경이 없으면 lint/build/test를 생략할 수 있다. v0.5.1 이후 기능 구현 버전은 아래를 기본 검증으로 둔다.

- frontend `npm.cmd run lint`
- frontend `npm.cmd run build`
- backend `.\gradlew.bat test`
- 권한별 200/403 수동 확인
- 감사 로그 기록 확인
- 재고 수량 정합성 확인
