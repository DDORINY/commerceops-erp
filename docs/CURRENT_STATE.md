# 현재 구현 현황

> 기준일: 2026-07-04  
> 판정 기준: 소스 파일 존재 여부가 아니라 컴파일, 정적 검사, 테스트와 사용자 흐름까지 포함한다.

## 요약

현재 저장소에는 쇼핑몰 핵심 기능과 Claude 로드맵의 배송·반품·문의·기초 회계 구현이 포함되어 있다. 2026-07-04 안정화 작업으로 백엔드 컴파일/테스트와 프론트 lint/build 차단 문제가 해결됐다. 추가 기능은 **구현 완료, 통합/E2E 검증 보강 중** 상태다.

### 검증 결과

| 검사 | 결과 | 판정 |
| --- | --- | --- |
| 백엔드 `gradlew test` | H2 테스트 프로필에서 16개 테스트, 실패 0 | ✅ 통과 |
| 프론트엔드 `npm run lint` | 오류 0, 경고 0 | ✅ 통과 |
| 프론트엔드 `npm run build` | 별도 검증 출력 디렉터리에서 23개 라우트 프로덕션 빌드 | ✅ 통과 |
| 자동화 테스트 | context + 회계 2 + 배송 2 + 반품 1 + 문의 1 + 창고/풀필먼트 6 + 입고 1 + 주문취소 2 | 🟡 핵심 서비스 확보, 통합/E2E 추가 필요 |

로컬 dev 서버가 `.next`를 점유한 상태에서도 검증할 수 있도록 `NEXT_DIST_DIR`를 지원한다.

## Claude 로드맵과 코드 차이

| 로드맵 제안 | 실제 코드 | 정리 방향 |
| --- | --- | --- |
| `PATCH /admin/shipments/{id}/status` | `PATCH /admin/shipments/{id}/deliver` | 현재 코드를 기준으로 명세 확정 또는 범용 상태 API로 재설계 |
| `POST /refunds/mock/complete` | 별도 환불 결제 API 없음. 반품 승인 시 상태/재고/회계만 변경 | 실제 환불 처리와 멱등성 설계 필요 |
| 회계 매출/환불 보고서 API | `/summary`, `/entries`만 존재 | v0.4에서 보고서와 정식 전표 구조 구현 |
| `account_codes`, `accounting_entry_lines` | `accounting_entries`만 존재 | v0.4 복식부기 범위로 이동 |
| 배송·반품·문의·회계가 향후 기능 | 관련 소스가 이미 추가됨 | v0.2 안정화 및 v0.3 완성 대상으로 재분류 |

## 지금 가장 먼저 할 일

1. 배송 → 반품 승인 → 원창고 복구 → 회계 기록을 실제 DB 통합 테스트로 검증한다.
2. 피킹·패킹·출고 작업 상태를 설계한다.
3. 공급사·발주·입고 검수 기능을 설계한다.
4. 실제 PG 도입을 위한 승인·취소·웹훅·대사 어댑터를 설계한다.
5. mock을 사용하는 고객 관리·상품 등록 임시 처리를 실제 API로 전환한다.

## 알려진 문서 부채

- 기존 `API.md`에는 아직 구현되지 않은 로그아웃, 고객 관리, 파일 업로드 API가 구현된 것처럼 섞여 있다.
- `DB_SCHEMA.md`에 추가 테이블 요약은 반영했지만 실제 DDL 마이그레이션과 인덱스 명세가 없다.
- 기존 구조 문서의 라우트·모듈 목록에는 최근 추가 기능이 빠져 있다.
- 운영 환경, DB 마이그레이션, CI/CD, 관측성, 백업/복구 문서가 아직 없다.

이 부채는 버전별 기능 구현과 동시에 고친다. 별도의 대규모 문서 정리만 기다리지 않는다.

## v0.1.2 업데이트 기록

- 관리자 리뷰 관리 화면 `/admin/reviews`를 추가했다.
- 관리자 리뷰 목록 조회는 `GET /api/admin/reviews`를 사용한다.
- 평점 필터와 상품명/작성자명/내용 검색을 지원한다.
- 관리자 리뷰 삭제는 `DELETE /api/admin/reviews/{reviewId}`를 사용한다.
- `/api/admin/reviews/**`는 `ADMIN`, `SUPER_ADMIN` 권한으로 제한한다.
- 리뷰 엔티티에 숨김 상태 컬럼은 없으므로 v0.1.2에서는 상태 변경 대신 삭제 흐름만 구현했다.

## v0.1.3 업데이트 기록

- 관리자 회계 관리 화면 `/admin/accounting`은 기존 `GET /api/admin/accounting/summary`, `GET /api/admin/accounting/entries`를 사용한다.
- 요약 카드는 실제 회계 기록 기준으로 총 매출, 총 환불, 총 입고 금액, 순매출을 표시한다.
- 회계 내역 목록은 `SALE`, `REFUND`, `INBOUND` 유형 필터와 페이지네이션을 지원한다.
- CSV 다운로드는 현재 화면에 조회된 회계 내역 페이지 기준으로 클라이언트에서 생성한다.
- `/api/admin/accounting/**`는 `ADMIN`, `SUPER_ADMIN` 권한으로 제한되어 있다.
- 정식 복식부기, 계정과목, 마감, 정산 보고서는 v0.2 이후 고도화 범위로 유지한다.

## v0.1.4 업데이트 기록

- 관리자 매출 통계 화면 `/admin/sales`는 실제 관리자 대시보드 API를 사용한다.
- 기간별 매출/주문 수는 `GET /api/admin/dashboard/sales`를 사용하며 `DAILY`, `MONTHLY` 기간 단위를 지원한다.
- 전체 요약 지표는 `GET /api/admin/dashboard/summary`를 사용한다.
- 인기 상품 TOP 10은 `GET /api/admin/dashboard/top-products`를 사용한다.
- 최근 결제 완료 주문은 `GET /api/admin/orders?status=PAID`를 사용한다.
- 화면에는 로딩, 부분 실패 에러, 데이터 없음 상태를 표시한다.
- 상품별 지표 기간 필터와 고급 BI/차트 라이브러리 도입은 v0.2 이후 범위로 유지한다.

## v0.1.5 업데이트 기록

- 관리자 창고 관리 화면 `/admin/warehouses`는 실제 창고 API를 사용한다.
- 창고 목록/등록은 `GET`, `POST /api/admin/warehouses`를 사용한다.
- 창고별 재고 조회와 미배정 재고 배치는 `GET /api/admin/warehouse-stocks`, `POST /api/admin/warehouse-stocks/allocate`를 사용한다.
- 재고 이동 목록/요청/완료 처리는 `GET`, `POST`, `PATCH /api/admin/stock-transfers/**`를 사용한다.
- 창고별 재고 화면은 창고 필터와 상품명 검색, 로딩/에러/빈 상태를 처리한다.
- `/api/admin/warehouses/**`, `/api/admin/warehouse-stocks/**`, `/api/admin/stock-transfers/**`는 `ADMIN`, `SUPER_ADMIN` 권한으로 명시 제한한다.
- 주문 결제 예약, 배송 출고, 주문 취소 예약 해제, 반품 원창고 복구 흐름은 기존 `WarehouseFulfillmentService` 기준을 따른다.
- 피킹/패킹/출고 자동화와 복잡한 WMS 정책은 v0.2 이후 고도화 범위로 유지한다.
