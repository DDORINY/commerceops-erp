## 구현 결정

- 로그아웃은 서버 세션이 없는 stateless JWT 구조를 유지한다.
- Redis 기반 블랙리스트는 v0.2.1 범위에서 제외한다.
- 백엔드 `POST /api/auth/logout`은 클라이언트와 API 계약을 맞추는 no-op 확인 API로 둔다.
- 실제 로그아웃 기준 동작은 프론트에서 `accessToken`, `refreshToken`, `user`를 삭제하고 로그인 화면으로 이동하는 것이다.
- Refresh Token은 별도 JWT이며 `type=refresh` claim을 가진다.
- v0.2.1 최소 구현에서는 Refresh Token을 `localStorage.refreshToken`에 저장한다.
- 운영 보안 수준의 HttpOnly cookie, 서버 저장소 기반 refresh token rotation, 탈취 감지, 블랙리스트는 후속 보안 고도화 범위로 분리한다.

## 보안 점검 결과

- `SecurityConfig` 기준 공개 API는 health, signup, login, refresh, logout, 공개 상품/카테고리 조회로 제한된다.
- `/api/admin/users/**`, `/api/admin/orders/**`, `/api/admin/dashboard/**`, `/api/admin/inventory/**`, `/api/admin/shipments/**`, `/api/admin/returns/**`, `/api/admin/inquiries/**`, `/api/admin/reviews/**`, `/api/admin/warehouses/**`, `/api/admin/warehouse-stocks/**`, `/api/admin/stock-transfers/**`, `/api/admin/products/**`, `/api/admin/coupons/**`, `/api/admin/accounting/**`는 `ADMIN`, `SUPER_ADMIN` 중심으로 제한된다.
- 프론트 `apiClient`는 401 발생 시 refresh token으로 1회 재발급을 시도하고 실패하면 토큰/사용자 정보를 제거한다.
- 백엔드 인증 필터는 `type=access` 토큰만 SecurityContext 인증에 사용한다. `type=refresh` 토큰은 `/api/auth/refresh` 재발급에만 사용한다.
- 관리자 레이아웃은 토큰이 없으면 로그인 화면으로 이동하고, 저장된 역할이 일반 사용자면 접근 거부 안내를 표시한다.
- CORS는 `http://localhost:3000`만 허용하고 있으며 wildcard origin은 사용하지 않는다.
- 콘솔에 토큰을 출력하는 코드 없음 확인.
