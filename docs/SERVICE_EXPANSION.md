# 서비스 확장 계획

이 문서는 Claude의 추가 기능 로드맵인 배송·반품·문의·회계·창고·인사·생산 이후를 포함해, CommerceOps ERP를 실제 운영 서비스와 장기 플랫폼으로 확장할 때 필요한 범위를 정리한다.

## 확장 순서

기능은 아래 층을 아래에서 위로 쌓는다. 하위 층이 불안정하면 상위 기능을 먼저 늘리지 않는다.

```text
5. 플랫폼/SaaS      멀티테넌시, 외부 연동, 국제화, 생태계
4. 데이터/자동화    BI, 예측, 이상 탐지, 워크플로 자동화
3. ERP 확장         조달, 창고, 회계, 인사, 생산
2. 커머스 운영      결제, 배송, 반품, CS, 프로모션
1. 서비스 기반      보안, 배포, 관측성, 백업, 개인정보
```

## 1. 상용 커머스 기능

### 상품과 가격

- 상품 옵션과 SKU, 옵션별 가격/재고/바코드
- 가격표, 기간 할인, 회원 등급 가격, 묶음 상품
- 디지털/정기구독/예약 상품 유형 선택 지원
- 상품 승인, 게시 예약, 변경 이력, 다국어 콘텐츠

### 주문과 결제

- 실제 PG 다중 수단, 간편결제, 가상계좌
- 부분 취소/부분 환불, 결제 실패 복구, 중복 요청 방지
- 세금계산서/현금영수증/영수증
- 주문 분할·합배송·부분 배송·백오더
- 비회원 주문과 선물 주문
- PG 정산 내역 대사와 차이 자동 탐지

### 마케팅과 고객

- 쿠폰, 프로모션, 포인트, 회원 등급
- 리뷰, Q&A, 찜, 재입고 알림
- 고객 세그먼트와 캠페인 동의 관리
- 유입/전환 퍼널, 장바구니 이탈 분석
- 고객 생애 가치, 재구매, 코호트 분석

### CS와 사후 처리

- 티켓 기반 고객 지원, 담당자·우선순위·SLA
- 반품 수거, 검수, 교환, 부분 환불
- 템플릿 답변과 대화 이력
- 분쟁/클레임과 증빙 파일
- 고객 알림 이력과 재발송

## 2. ERP 운영 확장

### 조달/공급사

- 공급사, 견적, 구매 요청, 발주, 입고, 매입 반품
- 공급사별 리드타임·최소 주문 수량·단가 이력
- 발주 승인과 예산 통제
- 매입채무와 지급 일정

### 창고/물류

- 다중 창고와 로케이션
- 재고 예약/가용/검수/불량/보류 수량 분리
- 피킹 웨이브, 패킹, 출고 라벨
- 택배사 API와 배송 추적 웹훅
- 로트/시리얼/유통기한과 선입선출
- 재고 실사, 순환 실사, 손실 분석

### 재무/회계

- 복식부기, 계정과목, 원장, 보조부
- 매출채권/매입채무, 정산, 비용, 자금
- 세금·부가세·통화·환율 정책
- 회계 기간 마감, 반제, 감사 추적
- 외부 회계/세무 서비스 연동

### 인사/생산

인사와 생산은 모든 커머스 사업자에게 필요한 것은 아니므로 코어에서 분리된 선택 모듈로 둔다.

- 인사: 조직, 직원, 근태, 휴가, 급여 기초, 접근 권한
- 생산: BOM, 공정, 작업 지시, 자재 소요, 완공, 불량, 원가

## 3. 데이터와 자동화

- 이벤트 수집과 운영 데이터 웨어하우스
- 실시간 운영 지표와 경영 BI
- 수요 예측과 자동 발주 제안
- 재고 부족/과잉/장기 재고 탐지
- 결제·환불·재고 이상 거래 탐지
- 고객 문의 분류와 답변 초안
- 승인·알림·배치 작업을 구성하는 워크플로 자동화
- 모든 자동 결정의 근거, 승인자, 결과를 남기는 감사 로그

AI 기능은 개인정보와 영업 데이터를 최소한으로 사용하고, 사람이 검토할 수 있으며, 잘못된 자동 실행을 되돌릴 수 있게 만든다.

## 4. 플랫폼/SaaS 확장

### 멀티테넌시

- 조직/상점별 데이터 격리
- 테넌트별 사용자·역할·설정·브랜딩
- 요금제, 사용량 계측, 구독 결제
- 테넌트별 데이터 내보내기와 삭제
- 대형 고객용 SSO/SAML과 감사 로그

### 외부 채널과 생태계

- 오픈마켓/자사몰 주문·상품·재고 동기화
- 택배, PG, 세무, 메시징, 회계 서비스 커넥터
- 공개 API, 웹훅, API 키/OAuth, 호출량 제한
- 파트너 앱 권한과 연동 상태 모니터링
- 실패 이벤트 재처리와 dead-letter queue

### 국제화

- 언어, 통화, 시간대, 주소 형식
- 국가별 세금·개인정보·전자상거래 정책
- 환율과 다중 원장 정책
- 데이터 지역성과 해외 이전 요구사항

## 5. 운영 서비스 필수 기능

눈에 보이는 제품 기능과 별개로 아래는 정식 서비스의 필수 범위다.

| 영역 | 필수 결과물 |
| --- | --- |
| 신뢰성 | SLO, 헬스 체크, 타임아웃/재시도, 장애 격리 |
| 관측성 | 구조화 로그, 메트릭, 트레이스, 알림 대시보드 |
| 배포 | CI/CD, 환경 분리, 마이그레이션, 롤백 |
| 데이터 보호 | 암호화, 백업, 복원 훈련, 보존/파기 |
| 보안 | 최소 권한, 비밀 관리, 감사 로그, 취약점 대응 |
| 고객 운영 | 문의·장애·환불 대응 절차와 SLA |
| 법무/정책 | 약관, 개인정보 처리, 쿠키/마케팅 동의, 고지 |
| 비용 | 인프라/외부 API 비용 지표와 예산 경보 |

법률·세무 요건은 출시 국가와 사업 형태에 따라 달라지므로 실제 출시 전 해당 분야 전문가의 검토를 받아야 한다.

## 장기 아키텍처 방향

- 당장은 모듈형 모놀리스로 도메인 경계를 명확히 유지한다.
- 결제·재고·회계 사이에는 멱등 이벤트와 outbox 패턴을 검토한다.
- 읽기 부하가 큰 검색·리포트는 운영 트랜잭션 조회와 분리한다.
- 파일은 애플리케이션 서버가 아니라 객체 저장소와 CDN을 사용한다.
- 도메인을 서비스로 분리하는 것은 팀 규모, 배포 독립성, 부하 요구가 실제로 생긴 뒤 결정한다.
- 모든 외부 연동은 어댑터, 타임아웃, 재시도, 서킷 브레이커, 재처리 큐를 갖춘다.

## 확장 기능 채택 질문

새 기능을 로드맵에 넣기 전에 다음을 답한다.

1. 어떤 사용자와 운영 문제를 해결하는가?
2. 성공을 어떤 지표로 측정하는가?
3. 기존 주문·재고·결제·회계 상태에 어떤 영향을 주는가?
4. 실패하면 돈·수량·개인정보에 어떤 문제가 생기는가?
5. 외부 서비스, 법률, 운영 인력이 필요한가?
6. 최소 범위와 이번 버전에서 하지 않을 것은 무엇인가?
7. 되돌리거나 비활성화할 수 있는가?
