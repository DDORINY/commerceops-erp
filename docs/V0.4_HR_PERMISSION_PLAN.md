# v0.4 HR and Permission Management Plan

## 목적

v0.4는 v0.3 상품/전시/운영관리 고도화 이후 관리자 조직 운영을 실무 ERP 관점으로 확장하는 단계다. 최고관리자가 직원, 부서, 직급, 직무/역할, 권한 그룹을 관리하고, 관리자 화면과 API 접근을 세분화된 권한 기준으로 통제할 수 있게 만드는 것을 목표로 한다.

## v0.4 큰 방향

- 현재 `USER`, `MANAGER`, `ADMIN`, `SUPER_ADMIN` 역할 기반 접근을 유지하되, 실무형 권한 그룹과 개별 권한으로 확장한다.
- 직원의 조직 정보와 계정 상태를 분리해 재직/휴직/퇴사, 활성/비활성 흐름을 관리한다.
- 관리자 사이드바와 API 권한 정책을 권한 그룹/개별 권한 기준으로 점진적으로 연결한다.
- 감사 로그를 리뷰/상품 중심에서 주요 관리자 작업 전체로 확장할 수 있는 구조로 정리한다.

## v0.4 세부 버전 계획

### v0.4.0 인사관리/권한관리 계획 수립

- 목적: v0.4 전체 범위와 제외 범위, DB/API/프론트 설계를 문서화한다.
- 포함 범위: 직원, 부서, 직급, 역할, 권한 그룹, 권한 매트릭스, 감사 로그 확장 계획.
- 제외 범위: 실제 기능 구현.
- 완료 조건: 계획 문서, 체크리스트, v0.4.x 작업 순서 확정.

### v0.4.1 직원/부서/직급 기본 모델

- 목적: 인사/권한관리의 최소 DB 기반을 만든다.
- 포함 범위: `departments`, `positions`, `staff_profiles` 모델과 Flyway migration.
- 제외 범위: 복잡한 조직도, 급여, 근태.
- 완료 조건: 기본 CRUD service/repository 기반과 문서 갱신.

### v0.4.2 직원 관리 화면/API

- 목적: 최고관리자가 직원 계정과 인사 프로필을 조회/관리할 수 있게 한다.
- 포함 범위: 직원 목록, 상세, 생성/수정, 활성/비활성, 재직/휴직/퇴사 상태.
- 제외 범위: SSO/OAuth, 대량 엑셀 업로드.
- 완료 조건: 관리자 직원 관리 화면과 API 검증.

### v0.4.3 권한 그룹/역할 관리

- 목적: 관리자 역할과 권한 그룹을 관리할 수 있게 한다.
- 포함 범위: 권한 그룹 생성/수정, 사용자 권한 그룹 할당, 기존 role과 병행 운영.
- 제외 범위: 메뉴/기능별 세부 매트릭스 완성.
- 완료 조건: 권한 그룹 DB/API/관리 화면 동작.

### v0.4.4 메뉴별/기능별 권한 매트릭스

- 목적: 메뉴 접근과 기능 실행 권한을 명시적으로 관리한다.
- 포함 범위: `permissions`, `permission_group_permissions`, `admin_menu_permissions`.
- 제외 범위: 복잡한 승인 워크플로우.
- 완료 조건: 권한 매트릭스 조회/수정과 문서화.

### v0.4.5 관리자 사이드바 권한 연동 고도화

- 목적: 현재 역할 기반 사이드바 노출을 권한 기반으로 고도화한다.
- 포함 범위: 메뉴별 권한 기준 노출, 비인가 메뉴 접근 UX, fallback 정책.
- 제외 범위: API 권한 세분화 전체.
- 완료 조건: 권한 그룹에 따라 관리자 메뉴가 달라짐.

### v0.4.6 API 권한 정책 세분화

- 목적: SecurityConfig와 서비스 권한 검증을 세분화된 permission 기준으로 정리한다.
- 포함 범위: 주요 관리자 API의 read/write/manage 권한 분리.
- 제외 범위: 외부 IAM/SSO.
- 완료 조건: 권한별 API 접근 테스트와 문서 갱신.

### v0.4.7 관리자 작업 이력/감사 로그 확장

- 목적: 상품/리뷰 외 주요 관리자 작업을 추적 가능하게 만든다.
- 포함 범위: 직원/권한/설정/주문/상품/쿠폰 등 주요 변경 이벤트 기록 기준.
- 제외 범위: 복잡한 검색/다운로드 감사 리포트.
- 완료 조건: 핵심 관리자 변경 작업의 감사 로그 기록.

### v0.4.8 사업자/약관 설정 저장과 버전관리

- 목적: v0.3.5.1에서 만든 설정 진입점을 실제 저장 구조로 연결한다.
- 포함 범위: 사업자 정보, 이용약관, 개인정보처리방침, 배송/반품 정책 저장과 버전 기록.
- 제외 범위: 전자서명, 법정 서식 자동 생성.
- 완료 조건: 설정 저장/조회/버전 이력 확인.

### v0.4.9 v0.4 통합 검증 및 v0.5 이관

- 목적: v0.4 인사관리/권한관리 범위를 통합 검증하고 v0.5 재고/생산/바코드로 이관한다.
- 포함 범위: 태그/체크리스트, API/DB/프론트/권한/감사 로그 검증.
- 제외 범위: 신규 기능 구현.
- 완료 조건: v0.4 통합 검증 문서와 v0.5 이관 후보 정리.

## DB 설계 후보

| 테이블 | 목적 | 핵심 필드 후보 | 관계 |
| --- | --- | --- | --- |
| `departments` | 부서 관리 | `id`, `name`, `code`, `parent_id`, `sort_order`, `active`, `created_at`, `updated_at` | self parent, staff와 1:N |
| `positions` | 직급/직책 관리 | `id`, `name`, `level`, `sort_order`, `active`, `created_at`, `updated_at` | staff와 1:N |
| `staff_profiles` | 사용자 계정의 직원 프로필 | `id`, `user_id`, `department_id`, `position_id`, `employee_no`, `employment_status`, `joined_at`, `left_at`, `active` | users와 1:1, department/position N:1 |
| `permission_groups` | 권한 그룹 | `id`, `name`, `description`, `system_group`, `active`, `created_at`, `updated_at` | permission, user와 N:M |
| `permissions` | 기능 권한 정의 | `id`, `code`, `name`, `domain`, `action`, `description`, `active` | permission_group과 N:M |
| `permission_group_permissions` | 권한 그룹-권한 매핑 | `permission_group_id`, `permission_id`, `created_at` | permission_groups와 permissions 연결 |
| `user_permission_groups` | 사용자-권한 그룹 매핑 | `user_id`, `permission_group_id`, `created_at`, `created_by` | users와 permission_groups 연결 |
| `admin_menu_permissions` | 관리자 메뉴별 필요 권한 | `id`, `menu_key`, `menu_path`, `required_permission_code`, `visible`, `created_at` | permissions 참조 후보 |
| `audit_logs` 확장 | 관리자 작업 이력 확장 | `ip_address`, `user_agent`, `request_method`, `request_path`, `before_json`, `after_json` 후보 | 기존 audit log 확장 |
| `admin_activity_logs` | 감사 로그 분리 후보 | `actor_id`, `action_type`, `target_type`, `target_id`, `summary`, `metadata_json`, `created_at` | 기존 audit log 대체/보완 후보 |

## 검증 기준

- 문서 계획 버전은 코드 변경이 없으면 lint/build/test를 생략할 수 있다.
- DB/API/프론트 코드가 변경되는 구현 버전은 frontend lint/build와 backend test를 실행한다.
- 권한 관련 구현 버전은 `SUPER_ADMIN`, `ADMIN`, `MANAGER`, 일반 사용자 기준 접근 정책을 체크리스트에 기록한다.
