# v0.4 HR and Permission Management Plan

기준 버전: `v0.4.0-hr-permission-plan`

문서 상태: v0.3.9 후보안을 v0.4.0에서 확정 계획으로 보강했다. v0.4.0은 문서 계획 버전이며 기능 코드, DB migration, API 구현은 포함하지 않는다.

## 목적

v0.4는 v0.3 상품/전시/운영관리 고도화 이후 관리자 조직 운영을 실무 ERP 관점으로 확장하는 단계다. 최고관리자가 직원, 부서, 직급, 직무/역할, 권한 그룹을 관리하고, 관리자 화면과 API 접근을 세분화된 권한 기준으로 통제할 수 있게 만드는 것을 목표로 한다.

## v0.4 큰 방향

- 현재 `USER`, `MANAGER`, `ADMIN`, `SUPER_ADMIN` 역할 기반 접근을 유지하되, 실무형 권한 그룹과 개별 권한으로 확장한다.
- 직원의 조직 정보와 계정 상태를 분리해 재직/휴직/퇴사, 활성/비활성 흐름을 관리한다.
- 관리자 사이드바와 API 권한 정책을 권한 그룹/개별 권한 기준으로 점진적으로 연결한다.
- 감사 로그를 리뷰/상품 중심에서 주요 관리자 작업 전체로 확장할 수 있는 구조로 정리한다.

## v0.4 확정 범위

- 직원 관리: 직원 프로필, 계정 연결, 재직/휴직/퇴사, 활성/비활성 상태.
- 부서 관리: 부서 코드, 상위 부서, 정렬, 활성 상태.
- 직급 관리: 직급명, 레벨, 정렬, 활성 상태.
- 직무/역할 관리: 기존 `UserRole`과 권한 그룹을 병행해 점진 전환.
- 권한 그룹 관리: 권한 코드 묶음, 사용자 할당, 시스템 기본 그룹.
- 메뉴별 권한: 관리자 사이드바 메뉴 노출과 직접 URL 접근 UX 기준.
- 기능별 권한: 조회, 생성/수정, 상태 변경, 환불, 마감 같은 실행 권한.
- 권한별 관리자 사이드바 노출: role 기반에서 permission 기반으로 점진 전환.
- 권한별 API 접근 제어: `SecurityConfig`의 role 정책을 유지하면서 service/pre-authorization 후보를 검토.
- 관리자 작업 이력/감사 로그 확장: 직원/권한/설정/주문/상품/쿠폰 등 주요 변경 작업 기록.
- 사업자/약관 설정 저장과 버전관리: v0.3.5.1 설정 진입점을 실제 저장 구조로 연결.

## v0.4 세부 버전 계획

### v0.4.0 인사관리/권한관리 계획 수립

- 목적: v0.4 전체 범위와 제외 범위, DB/API/프론트 설계를 문서화한다.
- 포함 범위: 직원, 부서, 직급, 역할, 권한 그룹, 권한 매트릭스, 감사 로그 확장 계획.
- 제외 범위: 실제 기능 구현.
- DB 변경 후보: 없음. 문서 계획만 수행.
- API 변경 후보: 없음. 후보 API만 문서화.
- 프론트 화면 변경 후보: 없음. 후보 화면만 문서화.
- 권한 정책: 기존 role 유지와 permission group 병행 도입 원칙 확정.
- 완료 조건: 계획 문서, 체크리스트, v0.4.x 작업 순서 확정.
- 검증 기준: 문서 변경만 확인하고 lint/build/test는 생략 가능.

### v0.4.1 직원/부서/직급 기본 모델

- 목적: 인사/권한관리의 최소 DB 기반을 만든다.
- 포함 범위: `departments`, `positions`, `staff_profiles` 모델과 Flyway migration.
- 제외 범위: 복잡한 조직도, 급여, 근태.
- 구현 상태: v0.4.1에서 `Department`, `Position`, `StaffProfile`, `EmploymentStatus`와 repository/service 기반을 추가했다.
- DB 변경: `V10__create_hr_staff_base.sql`에서 `departments`, `positions`, `staff_profiles`를 생성한다.
- API 변경: `GET /api/admin/hr/departments`, `GET /api/admin/hr/positions`, `GET /api/admin/hr/staff-profiles` 조회 API를 추가했다.
- 프론트 화면 변경: 없음. 직원 관리 화면은 v0.4.2 범위다.
- 권한 정책: 기존 role 기반을 유지하고 HR 조회 API는 `ADMIN`, `SUPER_ADMIN`만 접근한다. permission group 기반 접근 제어는 v0.4.3 이후로 이관한다.
- 완료 조건: 기본 repository/service, 조회 API, migration, 문서 갱신, backend test 통과.
- 검증 기준: backend test와 migration 문법 확인.

### v0.4.2 직원 관리 화면/API

- 목적: 최고관리자가 직원 계정과 인사 프로필을 조회/관리할 수 있게 한다.
- 포함 범위: 직원 목록, 상세, 생성/수정, 활성/비활성, 재직/휴직/퇴사 상태.
- 제외 범위: SSO/OAuth, 대량 엑셀 업로드.
- 구현 상태: v0.4.2에서 `/api/admin/staff/**` 직원 관리 API와 `/admin/settings/staff` 화면을 구현했다.
- DB 변경: 없음. v0.4.1의 `staff_profiles`, `departments`, `positions`, `users` 구조를 사용한다.
- API 변경: 직원 목록/상세/등록/수정/재직 상태 변경/활성 상태 변경 API를 추가했다.
- 프론트 화면 변경: `/admin/settings/staff`에서 목록, 필터, 등록/수정 폼, 상태 변경, 활성 토글을 제공한다.
- 권한 정책: 조회는 `ADMIN`, `SUPER_ADMIN`, 변경은 `SUPER_ADMIN`으로 제한한다. permission group 기반 권한은 v0.4.3 이후로 전환한다.
- 완료 조건: 관리자 직원 관리 화면과 API, 감사 로그 기록, lint/build/test 통과.
- 검증 기준: frontend lint/build, backend test, 직원 상태 변경 수동 확인 후보 기록.

### v0.4.3 권한 그룹/역할 관리

- 목적: 관리자 역할과 권한 그룹을 관리할 수 있게 한다.
- 포함 범위: 권한 그룹 생성/수정, 사용자 권한 그룹 할당, 기존 role과 병행 운영.
- 제외 범위: 메뉴/기능별 세부 매트릭스 완성.
- DB 변경 후보: `permission_groups`, `user_permission_groups`.
- API 변경 후보: `/api/admin/permission-groups/**`, `/api/admin/users/{userId}/permission-groups`.
- 프론트 화면 변경 후보: `/admin/settings/permission-groups`, `/admin/settings/roles`.
- 권한 정책: `ROLE_MANAGE` 중심, `SUPER_ADMIN`은 시스템 그룹 변경까지 허용.
- 완료 조건: 권한 그룹 DB/API/관리 화면 동작.
- 검증 기준: 권한 그룹 할당 후 사용자 권한 조회 결과 확인.

### v0.4.4 메뉴별/기능별 권한 매트릭스

- 목적: 메뉴 접근과 기능 실행 권한을 명시적으로 관리한다.
- 포함 범위: `permissions`, `permission_group_permissions`, `admin_menu_permissions`.
- 제외 범위: 복잡한 승인 워크플로우.
- DB 변경 후보: `permissions`, `permission_group_permissions`, `admin_menu_permissions`.
- API 변경 후보: `/api/admin/permissions`, `/api/admin/menu-permissions`.
- 프론트 화면 변경 후보: `/admin/settings/menu-permissions`.
- 권한 정책: `ROLE_MANAGE` 보유자만 권한 매트릭스 변경 가능.
- 완료 조건: 권한 매트릭스 조회/수정과 문서화.
- 검증 기준: 권한 그룹별 메뉴/기능 매핑 저장과 조회 확인.

### v0.4.5 관리자 사이드바 권한 연동 고도화

- 목적: 현재 역할 기반 사이드바 노출을 권한 기반으로 고도화한다.
- 포함 범위: 메뉴별 권한 기준 노출, 비인가 메뉴 접근 UX, fallback 정책.
- 제외 범위: API 권한 세분화 전체.
- DB 변경 후보: `admin_menu_permissions` 사용.
- API 변경 후보: 현재 사용자 권한/메뉴 조회 API 후보.
- 프론트 화면 변경 후보: `AdminSidebarV2`, 관리자 layout guard.
- 권한 정책: `SUPER_ADMIN` 전체 메뉴, permission code 기준 메뉴 노출, 기존 role fallback 유지.
- 완료 조건: 권한 그룹에 따라 관리자 메뉴가 달라짐.
- 검증 기준: `SUPER_ADMIN`, `ADMIN`, `MANAGER`, 제한 계정별 메뉴 노출 확인.

### v0.4.6 API 권한 정책 세분화

- 목적: SecurityConfig와 서비스 권한 검증을 세분화된 permission 기준으로 정리한다.
- 포함 범위: 주요 관리자 API의 read/write/manage 권한 분리.
- 제외 범위: 외부 IAM/SSO.
- DB 변경 후보: 없음 또는 permission 조회 캐시 후보.
- API 변경 후보: 주요 `/api/admin/**` 접근 정책 정리.
- 프론트 화면 변경 후보: 403 안내 UX 점검.
- 권한 정책: role은 1차 접근, permission code는 세부 기능 실행 기준으로 병행.
- 완료 조건: 권한별 API 접근 테스트와 문서 갱신.
- 검증 기준: 권한별 200/403 테스트와 기존 ADMIN/SUPER_ADMIN 흐름 회귀 확인.

## 권한 정책 설계

- 기존 `UserRole`은 유지한다.
- `SUPER_ADMIN`은 모든 permission code를 보유한 것으로 간주한다.
- `ADMIN`은 운영 변경 권한 중심의 기본 permission group을 부여하는 방향으로 설계한다.
- `MANAGER`는 조회 중심 permission group을 부여하는 방향으로 설계한다.
- 일반 `USER`는 관리자 메뉴/API 접근 권한을 갖지 않는다.
- permission group은 role을 대체하지 않고 v0.4에서 병행 도입한다.
- 관리자 사이드바는 기존 role fallback을 유지하면서 permission code 기반 노출로 점진 전환한다.
- API 접근 제어는 `SecurityConfig` role 제한을 유지한 뒤, 세부 기능은 permission code 검증을 추가하는 방식으로 확장한다.
- 권한 변경, 권한 그룹 할당, 직원 상태 변경은 audit log 기록 대상으로 본다.
